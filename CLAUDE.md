# The Music Quiz — project brief

A buzzer/quiz app for home quiz nights. **Single file: `index.html`** — plain HTML/CSS/JS, no build step, no framework. One page that renders different "views" from URL params.

- **Live:** https://jj-wcw.github.io/music-quiz/ · **Repo:** https://github.com/JJ-WCW/music-quiz (public)
- **Deploy = `git push` to `main`.** GitHub Pages rebuilds in ~40s. If a build hangs (>3 min "building"), re-trigger: `gh api -X POST repos/JJ-WCW/music-quiz/pages/builds`.
- **`./quiz-offline`** takes the public link down (404s) between quiz nights; **`./quiz-online`** brings it back. Pages can't be password-protected on this plan — see "Privacy" below.

## Views and URLs

`?view=home|join|player|display|admin` and `?bucket=<GAME_CODE>`.

- **admin** — the host's laptop dashboard (never cast). Password-gated.
- **display** — the projector tab (cast this). Lobby/QR, questions, banners, standings, podium.
- **player** — a phone: buzzer, answer buttons, text input, wager betting.
- **join** — sign-up (name, avatar, buzzer sound). **home** — landing.

## Backend

Firebase Realtime Database over plain REST (no SDK, no accounts):
`https://music-quiz-cac11-default-rtdb.europe-west1.firebasedatabase.app`

Everything for one game lives under `/games/<code>/`:

| Key | What |
|---|---|
| `qb-state` | the live game state (one object; see `defaultState()`) |
| `qb-players` | `{<playerId>: {id,name,avatar,score,buzzSound,...}}` (keyed, so joins are own-key writes) |
| `qb-rounds` | the quiz library (see below) |
| `qb-admin` | sha256 hex of the admin password |
| `qb-buzz-<epoch>/<playerId>` | one buzz |
| `qb-answer-<epoch>/<playerId>` | one question answer |
| `qb-ante-<epoch>/<playerId>` | one wager bet |

`kvGet` returns raw text → `JSON.parse`; `kvSet` PUTs a JSON string (Firebase parses the body and stores structured data). Game-code paths are **case-sensitive** — the app lowercases codes.

One path is **not** a game: `/games/qhistory/asked.json` is the cross-quiz **question history** — every question/song ever used, appended by the quiz-generation recipe (merge `PATCH`, never `PUT`) and checked before generating so quiz nights never repeat material. Schema and workflow live in `QUIZ-RECIPE.md`. The app itself never reads or writes it.

## The library: rounds are groups of questions

`qb-rounds` = `[...groups, ...suddenDeaths]`.

- **Group (a "round")**: `{id, type:'group', name:'General Knowledge', questions:[...]}`
- **Questions** (inside a group, any mix): `type:'song'` (layered music buzzer), `type:'question'` (4-way multiple choice), `type:'text'` (typed answer), `type:'slider'` (slide-to-guess a number)
- **Sudden death**: `type:'sudden'`, top level, **after** all groups. Sits outside the running order (no "Question X of Y", never counts as the final round).
- Legacy flat libraries are auto-folded into a single "Round 1" group by `upgradeLibrary()`.

**Full schema + how to bulk-generate a quiz: see `QUIZ-RECIPE.md`.** A fresh session can read that file alone and `curl` a whole quiz into a game code.

## Core mechanics

- **Song rounds**: instrument layers revealed one at a time, fewer layers = more points. First buzz locks everyone out. **One guess per song** (wrong = out for that song, buzzers reopen for everyone else). 15s to answer after buzzing.
- **Question start**: multiple-choice/text/slider rounds open with a **3-2-1 countdown** on the projector (`state.countdown`), so the question doesn't jump-scare the room. The host tab owns the clock: `loadRoundToPlay`/`closeBetting` set `countdown:true` with no `answerEpoch`, and `adminTick`'s `startCountedQuestion()` mints the epoch and `stampBuzzersOpened()` when it ends — so nothing is answerable or server-stamped until "GO". Song and sudden-death rounds have no countdown (they open the buzzers immediately).
- **Question/text rounds**: everyone answers on their phone against a per-round timer (`seconds`, default 15 / 25 for typing). No lockout. Fastest correct gets a bonus. **Multiple-choice and slider questions auto-end the moment every player has answered** — `adminAutoEnd` holds the `answerEpoch` already ended and `renderAdmin` calls `scoreQuestion` directly (set synchronously before the first `await`, so two overlapping polls can't double-score); what's left is the host poll noticing, ~2s end to end; text rounds never auto-end because typed answers may be revised until the whistle. **Text rounds go to a host review screen** — fuzzy matching (case/spacing/punctuation/accents ignored) pre-ticks near-misses, host approves with ✓/✗, nothing scores until "Confirm & score it".
- **Slider rounds** (`type:'slider'`): host sets min/max/answer; players drag on their phone. **Nearest guess wins** and ties genuinely share. Optional `sliderBand` — everyone within ±band also scores (leave 0 for cut-throat "guess the year"; set one for huge ranges like PS2 sales). A **bullseye** (`dist <= step/2`, i.e. the closest notch the slider can reach) takes the `bonus`; with a step of 1 that collapses to an exact hit. Scores itself — no review screen. The kitty still goes to one person: closest, fastest breaking a dead heat.
  - **The step is derived, not typed**: `niceStep()` = range/100 snapped to a 1/2/5×10ⁿ ladder, so 2m-10m slides in 100,000s and 1990-2025 slides in 1s. `sliderStepFor()` resolves it **once at load time** into `state.sliderStep`, so every view agrees; don't recompute it per view. A `sliderStep` override on the round wins.
  - **Number formatting is per-question**: `groupNums(sliderMax)` decides thousands separators, because "160,000,000" needs them and "2,007" is just wrong. Every `fmtNum`/`fmtGuess` call must pass it.
- **Powerups**: every player signs up with **5 powerups** (`PU_MAX_SLOTS`, duplicates allowed) from ✅ Auto-Correct (base points even if wrong), 💣 Bomb (pick a victim, −10 at reveal, **only fires if the bomber answers correctly**, wasted otherwise, no defending), ✂️ 50/50 (snips two wrong MC answers, MC-only — greyed elsewhere), 🎲 Double or Nothing (doubles what you win, burns on nothing), 🛢️ Oil Slick (5s of un-tappable dripping oil on a rival's phone, immediate), 🧨 Carpet Bomb (−5 off **every** rival at once, no aiming, the bomber flies above the blast, **only if they answer correctly**), 🦠 Virus (infects one rival: −3 now and −3 on **every** question after, until the victim answers one correctly and is cured; **only infects if the infector answers correctly**), 🥷 Ninja (swipes −`NINJA_STEAL` (3) off **every** unshielded rival and hands the lot to the player, **only if they answer correctly**), 🪖 Royal Marine (target a rival and lift up to `MARINE_STEAL` (2) of their held powerups into your own bank, respecting the 5-slot cap, **only if you answer correctly**), 🛡️ Shield (`immediate:true` — activates the instant it's equipped, **no correct answer needed**: `p.shield = SHIELD_QUESTIONS` (3) of **full immunity** to everyone else's powerups, decremented at the end of each question round). **Shields resolve FIRST in `applyPowerups`** (before virus ticks and every offensive play) so protection is live for the same question they're equipped: a shielded target makes bomb/virus/marine bounce (`ev.shielded`), is skipped by carpet/ninja (`ev.shieldedOut`), pauses an existing virus tick, and deflects an oil drip (`shieldedTarget(target, from)` gates all of it; a player's own powerup never blocked by their own shield). **The one thing a shield does NOT stop is the nuke** — see below. A shield turns the whole score badge **blue** (`.shielded`, priority over the green infected tint), announced live on the projector the moment it's raised (like oil) with a blue banner + `showShieldRaised` wash on the equipper's phone. The infection lives on `p.virus = {by, byName}` and bites at the **top** of `applyPowerups`, before new plays land, so a player infected this question takes one hit and not two; a correct answer cures instead of docking. Every bite gets its own green announcement — a breathing banner on the projector and a full-screen wash of drifting germs on the victim's phone (`showVirusHit`, `pointer-events:none` since it's an announcement, not a handicap) — plus a standing `qb-infected-badge` and a 🦠 beside their name on every board until they shake it off. **☢️ Tactical Nuke is NOT one of the choices** — every player is issued exactly **one for the whole quiz**, held on its own `p.nuke` boolean outside the slots, never re-earned (only a game reset reissues it). It's −20 to every rival, −10 to self, **nobody wins the question's points**, and it only detonates if the nuker answers correctly. **It is the great leveller and tears through EVERYTHING**: shields give no protection (the full −20 lands and the shield is destroyed, `ev.shieldsDestroyed`) and every standing infection on the board is wiped (`ev.virusesWiped`) — so a runaway leader can't turtle behind a shield, and the nuke resets all status effects to zero. Marked `granted:true` in `POWERUPS`; `PU_CHOOSABLE` is what the join/claim pickers offer, and `granted` powerups are refused by `claimPowerup`/`foldPuPicks` and consumed from `p.nuke` (not the slots) in `applyPowerups`, where they resolve LAST so they claw back auto/double grants, restate bomb maths, and shatter shields/viruses after every other play has settled. It was a normal pick once and everybody hoarded it, which turned the night into a warzone — one-shot keeps it a last-ditch leveller. **Question rounds only** (MC/typed/slider), playable from the 3-2-1 countdown to the whistle, **one play per player per question**, consumed win or lose. Several dramatic things can land on one reveal (a bomb *and* a virus biting), so the projector **queues** its banners loudest-first at 3.6s apart rather than letting one silently replace the rest. The projector announces oil slicks live (dripping banner + splat) and an armed nuke immediately (fixed flashing `qb-nuke-alert` strip + air-raid siren — imperative on `<body>` like the banners, so the poll can't blink it); bombs stay secret until the reveal. Plays are own-key writes under `qb-power-<puEpoch>` (`puEpoch` minted at load, NOT `answerEpoch`, so countdown plays work); the host resolves everything in `applyPowerups` (inside scoring, after normal points, so double includes bonuses and the bomb maths — `targetWon − 10 = net` — is spelled out in `puEvents` for the reveal). Picks are earned by **climbing**, not by a high-water mark: `refreshPuEarned` keeps a `p.puFloor` that **follows a sinking score down**, and every `PU_EVERY` (20) points gained above it banks one in `p.puOwed`. So a bombed player who fights back up 20 earns another — the catch-up mechanic helping the player it's meant to. (It used to track the highest rung ever reached, which punished exactly the player who'd just been knocked down: they had to re-climb to their old peak before earning anything.) `p.puLadder` records the spacing in force; when it differs (a changed `PU_EVERY`, or a player who joined before powerups existed) `refreshPuEarned` **rebases** — genuinely owed picks carry over and the floor restarts at the current score, so points already scored can't retroactively mint a backlog. The phone writes the choice to `qb-pupick/<id>` and the host folds it in (`foldPuPicks`) when a slot (max 5) is free. A game reset restores `puChosen` (the sign-up picks) and reissues everyone's nuke. The projector shows held icons beside names — an unspent nuke included — and the admin sees armed plays live. An **infection is not shown in that icon strip** (a 🦠 there read as "they hold a virus to play"); the player's whole score badge turns green instead. Playing ✅ Auto-Correct **counts as answering** — `answeredList()` merges auto plays into the answered chips and the everyone's-in check, or the question sat waiting for a tap that was never coming. Names cap at `NAME_MAX` (10), enforced on input and on save, because long names wreck the badge layout.
- **Wagers are DORMANT, not deleted** — the whole kitty/betting flow below still works and `isWager` questions still run; powerups replaced them as the levelling mechanic, so nothing sets `isWager` any more (the recipe stopped generating them). Reintroducing wagers is zero code: tick "wager round" on a question.
- **Wagers**: set `isWager` + `wagerClue` on **any** question type. Projector shows the clue first; players bet up to `MAX_BET` (10) into a kitty, **scores may go negative**. Players who don't fancy it tap **No bet**, which writes a `{amount:0, pass:true}` ante — that's what makes `antesDecided()` reach the player count so the host knows to stop waiting. Host closes betting, question plays normally, **the best-placed answerer who actually staked takes the whole kitty** — `settleKitty()`/`markCorrect()` filter to `stakedIn()`, so a non-bettor who was closer or faster still wins the normal points but **can't touch the pot** (it rolls over instead). If **exactly one player bet and that player wins**, the house matches their stake on top of the pot (`loneBetBonus()` — otherwise a solo bettor just wins their own points back, which made betting alone pointless); `state.betMatched` carries the matched amount to the reveal messaging. Betting is optional; everyone keeps their buzzer.
- **The kitty rolls over**: nobody wins it → `rollKittyOver()` banks the pot in `state.carryKitty` instead of burning it, and it survives round loads (like `quizTitle`) until someone finally wins, growing each time. `kittyTotal(s)` = this round's stakes **+** the carry — use it everywhere rather than summing `bets` by hand. A win zeroes `carryKitty`; `defaultState()` means a reset clears it too.
- **Ties**: when the last question of the **last** group reveals with the top level, admin shows a gold tie panel with one-click "Run sudden death". Loading a sudden death during a tie snapshots the tied leaders into `sdEligible` — **only their buzzers work**. The podium button soft-confirms over a tie (never hard-blocks).
- **Podium**: staged 3rd/2nd/1st reveal with a synthesized track. Host-triggered.

## Non-obvious invariants — don't break these

1. **Race-free writes.** Buzzes/answers/bets each write **only their own key** under an epoch, stamped with Firebase's server clock (`{'.sv':'timestamp'}`); the earliest timestamp wins. **Never** read-modify-write a shared array from a player device. `qb-players` is an **object keyed by player id**, never a whole-document read-modify-write. Sign-up `PUT`s only that player's key and the host `PATCH`es the players it knows about, so a phone joining can't erase the host's scoring and vice versa. (It used to be an array: a rejoining phone read the roster, appended itself and wrote it all back — spent powerups and nukes came back, consumed ones vanished, scores reverted. Removal is an explicit `DELETE` of the key, since a merge can't express "gone". Old array-shaped games are re-keyed on first read.) Beyond that, `qb-players` is written **only by the host tab** — single writer. **But one tab is not one writer:** the host mutates that array from several places at once (scoring, the poll's powerup fold, manual score edits), and two overlapping read-modify-write cycles silently clobbered each other — that's what erased banked powerup picks and re-offered them every question. Every host-side change now goes through **`withPlayers(mutate)`**, which serialises read→change→write into one critical section. **Never call `savePlayers` directly from host code** — use `withPlayers`, and never hold a players array across an `await` you didn't get from it.
2. **Server time only.** Reaction/answer times compare `ts` to `state.buzzersOpenedAt`, both server-stamped. Never trust phone clocks.
3. **Polls rebuild the DOM** (display 900ms, player 800ms, admin 1200ms) — this has caused several bugs:
   - Don't rebuild anything with an `<img>` or an input every poll. Rebuilding restarts the image load forever (so it never appears) and throws away a half-dragged slider or half-typed answer.
   - **Display**: live questions **render once** and only patch the answered-chips (`data-qepoch` + `#qb-answered-row`).
   - **Player**: every render fingerprints everything that decides the screen's *shape* (`prevPlayerShape`). Same shape → patch the live numbers by id (`#qb-p-score`, `#qb-p-kitty`, `#qb-p-pot`, `#qb-p-decided`, `#qb-p-answered`, `#qb-p-scoreboard`) and return. **Anything new that changes the phone's structure must go in that fingerprint**, or it'll never appear.
   - The player poll deliberately has **no `document.activeElement` guard**: the fingerprint already protects inputs, and skipping the poll on focus would strand a phone (iOS doesn't move focus when you tap a button) on a dead question forever. Don't "fix" this by re-adding the guard.
   - Admin preserves per-panel scroll across repaints via `data-scrollkey`.
   - Don't poll at all when there's no `bucket` (the setup screen) — it stole focus.
   - Admin form inputs that feed a live-recalculated hint (the slider's step/notches note) bind `onchange`, **not** `oninput` — re-rendering mid-keystroke steals focus. Never give such an input a `data-action`; `bindActions` wires those to `onclick`, so clicking into the field would re-render it out from under the cursor.
4. **Countdowns**: bake the *current* remaining value into each render; never reset the timer on a transient failed fetch (only a new epoch/reveal clears it).
5. **Stale renders**: every polled view takes a sequence ticket (`displaySeq`/`playerSeq`/`adminSeq`) and abandons itself if superseded.
6. **Audio**: Web Audio is synthesized (no files). `playFx` queues on `AudioContext.resume()` so the first tap is audible; a keepalive resumes it if the browser suspends it. iOS silent switch still mutes phones — the lobby tells players to unmute. Locking in any answer (MC/text/slider) plays that player's own buzzer sound (`playBuzzFor`), *before* the network write so it feels instant; reveal plays a correct ding / wrong jingle on each phone for their own result only — room-wide noise belongs to the display.

## Working conventions

- **Always syntax-check before shipping**: extract the inline `<script>` and run it through `new Function(src)` (JSC via `osascript -l JavaScript`).
- **Test against the real database** with a throwaway game code, then **delete the test game** (`curl -X DELETE .../games/<code>.json`). Screenshot visual changes.
- Admin is desktop-only by design (fixed-height dashboard, columns scroll individually). Projector/phones get the vibrant themed look; admin stays plain.
- Visual cues (played ✓, ▶ NOW) must **never disable** a button — the host may always redo anything.
- Confirm before destructive/irreversible host actions (loading over a live round, removing a player, resetting).

## Privacy / security posture

The Pages site is public (GitHub can't password-protect it on this plan; private access control is Enterprise-only). The Firebase DB is **open read/write under `/games/<code>`** by design — you can't *list* games, but anyone with the DB URL + a specific code can read/write that game, bypassing the client-side-only admin password. Fine for a home quiz. Use random auto-generated codes (not memorable ones) if you want the questions kept secret until the night. `./quiz-offline` hides the *app*, not the *data*.

## Known limitations

- Polling (~1s), not websockets — fine for a living room, not for split-second precision.
- Players who clear their phone's storage lose their identity (localStorage resume + "that's me" claim covers refreshes).
- Firebase free tier is far beyond what a living room needs.
